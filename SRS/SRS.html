1. Introduction

1.1 Purpose

The purpose of this product is to reduce cluster from government hospitals by allowing users to book their serials from home. This is a preliminary version of this product. The SRS will describe how the application will achieve this goal, brief discussion on how the system will work and detailed description about the product’s functionality.

1.2 Document Conventions

This SRS follows the IEEE standards for writing Software Requirements Specification. According to that conventions all texts are written in Times New Roman fonts and all heading are written in Bold formate.

1.3 Intended Audience and Reading Suggestions

This document is intended for project manager, designers, developers, testers and as well as the client. The overall description sections contains brief discussion of the product for the project manager and the client to give them some idea on how it will work. The rest of the document contains details on each and every functionality of the product which will help the designers to create necessary models and the developers to start building the application. The later parts will also help the testers to determine whether the requirements of this products are met or not.

1.4 Product Scope

For this stage, the main focus is to build a system which will let the patient book serials for doctor’s appointment, check their appointment status, search and view necessary information. Doctors will also be able to use is this app to check appointment schedules and give feedbacks to patients. Assistant can manage appointments and send medical report to patient through this app. As the product is aimed for the general people, the user interface will be very simple and user-friendly and in Bengali.

2. Overall Description

2.1 Product Perspective

The patients in government hospital has to stand in a long line to book an appointment with a doctor and even if they get a booking, the timing are never specified correctly. As a result they have to needlessly wait for an uncertain amount of time in the hospital which creates a big crowd. To resolve this issue we are designing this product which will let the users book their appointments without coming to the hospital.

2.2 Product Functions

The major functions of the product are stated blow:

  • User will be able to make an appointment of his desired doctor
  • User will be able to check their appointment status.
  • User will be able to view serial number, room number, timing of appointment and doctor’s details.
  • System will notify users on appointment confirmations and changes with necessary information.
  • Admins and doctors will be able to confirm/deny bookings and manage timing of appointments.
  • Doctors can give feedbacks (with medical report if any) to patients.
  • Admin will be able make changes to serials.

2.3 User Classes and Characteristics

  • Book Appointment:> Highly important. Used by Patient actor. Take input from Patients and asks for confirmation from admin before generating an appointment serial.
  • Serial Management: Highly important. Updates and manages serials in database after new confirmation, cancelation or changes to any appointment.
  • Send Notification: Important. Sends the user a notification for appointment confirmations and changes with necessary information.
  • View Request: Less important. Displays information requested by a user.
  • Send Report: Less important. Takes medical report as input from doctor and sends it to the specified Patient.

2.4 Operating Environment

The application will run on both Android and Windows. On Android the OS needs to be Android 5 or above. To run on Windows, the system requires a web browser such as Google Chrome, Microsoft Edge, Opera etc.

2.5 Design and Implementation Constraints

  • Corporate or regulatory policies: Personal Information of doctors and patients are essential for this app. Such information needs to be collected following to proper policies.
  • Language requirements: The user interface should come in both Bangla and English.
  • Specific Technologies: App needs a constant internet connections to keep all users updated.
  • Security Considerations: Encryption should be applied to ensure secured use.
  • Communications Protocols: HTTP.
  • Design conventions: Command and Query Responsibility Segregation.
  • Maintaining: Hospital administration will be in charge of maintaining.

2.6 User Documentation

A simple user manual will be provided with the product for the user to help them understand how to use this application.

2.7 Assumptions and Dependencies

  • Hospital authorities may decide to admit patients without a prior booking (may be in case of emergency) which may affect the serial of other patients shown by the app.
  • Doctors may take unaccounted break which may affect the timing provided by the app.
  • Manual booking will still be allowed. Such booking needs to be registered by hospital managements otherwise the system won’t be able to maintain or update the serials.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

  • Search features should be prominent and user friendly to use. The search results displayed in the list should be up to date and easy to understand.
  • View Serial Status should show the status in real time.
  • If the system loses the connection to the Internet, the user should be informed about any changes as soon as he gets back online.

5.2 Safety Requirements

In any case of extensive damage to a wide portion of the database due to catastrophic failure, such as a disk crash, the recovery method restores a past copy of the database that was backed up to archival storage and then reconstructs a current state by collecting data from the backed up log, up to the time of failure.

5.3 Security Requirements

The security of the communication between the system and server will be maintained carefully. Separate user name and mobile number should be needed for a user to create a new id.

5.4 Software Quality Attributes

  • Availability: The patients book ticket 24/7 by the app. Doctors should be available on the promised time
  • Maintainability: The system will maintain and ensure proper services for the patients.

5.4 Business Rules

There should be an emergency and a non-emergency case for the patients. It will ensure priority base health service. The hospital authority may still book serials manually for those who don’t have any technological devices.
